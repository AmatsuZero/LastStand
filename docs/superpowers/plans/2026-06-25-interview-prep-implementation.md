```markdown
# 3D 球场（CocosTennis）— 面试叙述稿

## 第一段：背景与决策（30秒）

腾讯体育有大量非版权网球赛事，图文直播用户体验很差，但买版权成本太高。
我的方案是用 3D 动画直播平替图文直播。

关键技术决策：
- 引擎选 Cocos Creator 而非 Unity：轻量、支持 Web 导出、团队学习成本低
- 3D 资产用 AI 文生方案：降低美术资源生产成本
- 数据源用 Opta 实时数据：保证比赛数据准确性和时效性

## 第二段：关键架构设计（1分钟）

核心挑战是"数据驱动"——Opta 实时数据如何驱动 3D 球员行动和球路？

我设计的链路：
```
Opta 数据 → 数据解析层（标准化） → 动作映射层（数据→动画指令） → 3D 渲染层（Cocos Creator 播放）
```

关键设计点：
1. 动作映射层的抽象：比赛事件（发球/击球/得分）映射为动画状态机，支持不同球员的动作配置
2. 延迟控制：数据到达后必须在 500ms 内完成渲染，通过预加载动画资源和帧同步保证
3. 弹幕引擎定制：3D 场景内的弹幕需要和传统直播弹幕体验一致，我基于 Cocos 的渲染管线做了定制

备选方案及为什么不选：
- 用 Unity：太重，Web 导出体积大，不适合轻量场景
- 用视频流模拟 3D：交互性差，无法支持弹幕和视角切换

## 第三段：推动落地与结果（30秒）

这是我全链路 Owner 的项目：
- 协调方：服务端（数据接口）、数据组（Opta 接入）、客户端（Cocos 集成）
- 交付节奏：先做离线 Demo 验证数据驱动链路 → 接入服务端实时数据 → 客户端工程化优化
- 结果：交付了可用的非版权赛事 3D 直播方案，填补了"无版权赛事直播体验"的空白

## 追问预案

**Q: 最大的技术难点是什么？**
A: 数据驱动链路的实时性。Opta 数据不是为 3D 动画设计的，字段映射有大量 corner case。我的做法是先建立数据→动画的标准映射表，再通过离线回放大量历史数据做测试，逐步完善映射规则。

**Q: 如果重做，你会改什么？**
A: 3D 资产生产流程。AI 文生方案虽然降低成本，但质量不稳定。如果重来，我会建立一套标准化的资产生产管线，AI 辅助 + 人工审核，而不是纯 AI 生成。

**Q: 团队协作中遇到过什么冲突？**
A: 数据组提供的 Opta 接口字段频繁变更，导致映射层反复修改。我的解决方式是推动建立接口契约文档，约定字段稳定性承诺，同时在映射层加了一层适配器隔离变更影响。
```

```markdown
# PC 开播助手（PCHelper）— 面试叙述稿

## 第一段：背景与决策（30秒）

业务痛点：
- 移动端开播屏幕小、拍摄角度受限，主播体验差
- 传统 PC 开播需要 OBS 运营人员配合，人力成本高

我的决策：
- 选 Electron + OBS Node 架构：Electron 保证跨平台和快速迭代（Vue.js 写 UI），OBS Node 复用 OBS 成熟的推流能力
- 关键创新：改造 OBS Node 支持 TRTC 推流——只修改协议类型（rtmp:// vs trtc://），最小化源码侵入，复用 OBS 原有 API

## 第二段：关键架构设计（1分钟）

最难的工程问题：跨进程视频渲染。

问题：Electron 的 Renderer 进程需要预览 OBS 画面，但 OBS 运行在 Main 进程。两个进程隔离，怎么把画面传过来？

我评估了三种方案：
1. **IPC 传帧**：把 Main 进程的 OBS 画面通过 IPC 逐帧传给 Renderer → 性能差，1080p 60fps 根本扛不住
2. **离屏渲染 + 共享纹理**：用 GPU 共享纹理跨进程 → 复杂度高，Electron 下兼容性差
3. **挖洞 + 同屏渲染**（我选的）：Renderer 进程在页面上预留一个透明区域（"挖洞"），Main 进程在这个区域上用原生窗口 + SDL 直接绘制 OBS 预览画面 → 零拷贝 GPU 渲染，性能最优

我的核心判断：宁可增加 C++/SDL 的复杂度，也不能牺牲视频预览的性能。这个方案需要处理 DPI 适配、渲染线程隔离、窗口生命周期管理等细节，但最终实现了流畅的 60fps 预览。

另一个关键设计：**SubRoom 多房间架构**
- 主播同时处于主直播间和虚拟陪看房间
- 通过 SEI（视频补充增强信息）将控制信息绑定到视频时间轴，做状态同步
- 保证了陪看房间的状态和主直播间严格同步

## 第三段：推动落地与结果（30秒）

从 0 到上线约 2 个月，涉及的技术栈：
- 前端：Electron + Vue.js
- 音视频：OBS Node C++ 改造 + TRTC 推流链路
- 原生渲染：SDL + 原生窗口管理

跨技术栈协调的挑战：前端、C++、音视频三个领域的人各自关注不同。我做了两件事：
1. 先画一张端到端架构图，让大家对"画面从哪来到哪去"有共识
2. 用最小可用版本（能推流能预览）快速跑通链路，再逐步完善功能

结果：项目上线后主播可以自主完成 PC 开播，不再依赖 OBS 运营人员。这是我最有成就感的项目，因为它从零开始，涉及多个技术领域的深度整合。

## 追问预案

**Q: 为什么不用 OBS 原生的虚拟摄像头方案？**
A: 虚拟摄像头方案无法做定制化——弹幕叠加、美颜滤镜、多场景切换这些需求都需要在 OBS 基础上做二次开发。OBS Node 给了我们完整的程序化控制能力。

**Q: Electron + 原生渲染的坑有哪些？**
A: 最大的坑是窗口层级管理。Electron 的透明窗口在 macOS 和 Windows 上行为不一致，macOS 上原生子窗口的 z-order 需要额外处理。另一个是 DPI 适配——高分屏下 SDL 渲染的坐标需要乘以 scale factor，但不同系统的 DPI 行为又不同。这些都是踩坑踩出来的。

**Q: 2 个月交付，中间有没有延期的风险？**
A: 有。OBS Node 的 TRTC 协议改造比预期复杂，因为 TRTC 的握手流程和 RTMP 差异大。我的做法是先做最小可用版本（能推流能预览），验证核心链路可行，再补周边功能。这样即使延期，核心价值也已经交付了。
```

```markdown
# AIRead 语音播报 SDK（KMMAIReadPractice）— 面试叙述稿

## 第一段：背景与决策（30秒）

业务需求：腾讯体育需要一个跨端（Android/iOS）的语音播报 SDK，让用户在比赛详情页听到 AI 语音播报。

我的技术决策：选 KMM（Kotlin Multiplatform）作为跨端方案。

为什么不选 Flutter/RN：
- SDK 体积敏感：Flutter/RN 会引入完整的引擎，包体积大
- KMM 编译为原生库：Android 是 .so，iOS 是 .framework，体积小且性能好
- 团队有 Kotlin 基础：学习成本可控

核心设计原则：业务逻辑收敛在 commonMain，平台能力下沉到平台层。

## 第二段：关键架构设计（1分钟）

核心架构：
```
commonMain（状态机/编排）
  ├── androidMain（ExoPlayer）
  └── iosMain（AVPlayer + QueuePlayer）
```

设计决策：

1. **状态机统一在 commonMain**：播放器的所有状态（idle/loading/playing/paused/error）和状态转换逻辑只维护一套。这是跨端 SDK 最核心的设计原则——避免双端状态漂移。

2. **播放器能力下沉平台层**：定义统一的 PlayerController 接口，Android 用 ExoPlayer 实现，iOS 用 AVPlayer + QueuePlayer 实现。commonMain 不关心具体播放器实现，只通过接口调用。

3. **WebSocket 流式接收 + 句子高亮闭环**：
```
WebSocket → AudioInfo（流式接收）
  → AIReadManager（编排：解析+调度）
  → PlayerController（播放）
  → WebViewNotifier（句子高亮同步到 WebView）
```
形成"听到哪看到哪"的用户体验闭环。

4. **重试策略**：支持 FIXED 和 MULTIPLICATION 两种延迟模式，应对网络波动。

## 第三段：推动落地与结果（30秒）

从 0 交付了完整的跨端语音播报 SDK，Android/iOS 双端对齐。

关键收获：
- 深刻理解了跨端 SDK 的核心命题：什么逻辑放 common 层、什么放平台层
- 接口契约设计的重要性：commonMain 和平台层之间的接口一旦定义好，双端可以独立开发
- 这个经验让我对跨端架构有了系统性的判断力，后面做技术选型时能更准确地评估方案

## 追问预案

**Q: commonMain 和平台层的边界怎么划？**
A: 我的原则是——有业务语义的逻辑放 commonMain，纯平台能力放平台层。比如"播放完句子A后播句子B"是业务逻辑，放 commonMain。"如何调用系统播放器播一个音频文件"是平台能力，放平台层。判断标准：如果这个逻辑换了平台实现会变，就放平台层。

**Q: 双端开发过程中遇到过什么不一致？**
A: 播放器行为不一致。ExoPlayer 和 AVPlayer 在某些边界状态（如快速切换音频）下行为有差异。我的解法是在 PlayerController 接口层做适配，把平台差异封装在实现内部，commonMain 看到的行为是统一的。

**Q: 如果现在重新设计，会有什么不同？**
A: 我会在协议层加版本字段和消息类型字段。当前协议设计比较简单，扩展性不够。另外 WebView 的 scrollToPosition 还没落地，这个需要补上。
```

```markdown
# 高频问题回答汇总

## 技术决策类

### Q1: 从零搭客户端团队，技术栈怎么选？

分三层看：
- **业务形态决定上层**：Native（高性能场景）、跨平台（多端一致）、小程序（流量入口）
- **团队能力决定中层**：KMM（有 Kotlin 基础）、Flutter（追求 UI 一致性）、RN（生态成熟）
- **基础设施决定底层**：CI/CD、监控体系、发布流程

我的经验矩阵：KMM 适合已有原生团队的跨端 SDK 场景，Electron 适合桌面端快速迭代，Cocos 适合 3D/游戏化场景。选型不是选"最好"的，是选"最匹配"的。

### Q2: 做过最难的架构决策？

PC 开播助手的跨进程视频渲染。Electron 下 Main 和 Renderer 进程隔离，OBS 画面在 Main 进程但需要在 Renderer 展示。

三个方案：
1. IPC 传帧：性能差，1080p 60fps 扛不住 → 不选
2. 离屏渲染 + 共享纹理：复杂度高，Electron 兼容性差 → 不选
3. 挖洞 + 同屏渲染：零拷贝 GPU 渲染，性能最优 → 选了

核心判断：宁可增加 C++/SDL 复杂度，不能牺牲视频预览性能。

### Q3: 跨平台方案怎么选？

三个维度：
- 性能要求：高性能 → Native / KMM；一般 → Flutter / RN
- 团队技能：Kotlin 基础 → KMM；Dart → Flutter；JS → RN
- SDK/库生态：特定平台能力（如 ARKit）→ Native

我的实践：AIRead SDK 选 KMM 因为体积敏感 + 团队有 Kotlin 基础。如果是业务 App 且追求 UI 一致性，Flutter 更好。没有银弹。

## 管理/推动类

### Q4: 项目延期怎么办？

三步走：
1. **判断根因**：需求变更 / 技术难题 / 人力不足 / 评估失误
2. **制定方案**：砍范围（降级非核心功能）/ 加人力 / 调排期
3. **透明沟通**：第一时间同步上级和相关方，不藏着掖着

我的实践：PC 开播助手 2 个月交付，OBS Node 改造比预期难。我用了最小可用版本策略——先验证推流+预览核心链路，再补周边功能。保证核心价值按时交付。

### Q5: 怎么处理技术分歧？

让数据和技术事实说话：
1. 列出方案 A 和方案 B 的优缺点、适用场景、风险
2. 如果有可量化的指标（性能、开发成本），用数据对比
3. 如果仍无法达成一致，我来做最终决策并对结果负责
4. 决策记录在文档里，后续可以复盘

我的博客习惯也体现了这一点——写下来，让事实可追溯。

### Q6: 你怎么做 1on1？

频率：两周一次，30 分钟。不谈进度（那是站会的事），谈三个问题：
1. 最近有什么让你开心/沮丧的事？
2. 遇到什么困难需要我帮忙？
3. 你想往什么方向成长？我能帮你做什么？

核心：1on1 是"他的时间"，不是"我的时间"。我听为主，说为辅。

## 业务 Sense 类

### Q7: 客户端团队怎么衡量价值？

三个维度：
- **业务指标**：留存、转化、用户时长——客户端直接影响用户体验
- **质量指标**：crash 率、ANR 率、启动时间、包体积——技术质量兜底
- **效率指标**：需求交付周期、发版频率——团队工程效能

我会定期向业务方同步这些数据，用数字说话，让��术价值可量化。

### Q8: 技术需求 vs 业务需求冲突？

先对齐优先级：业务需求通常更高优先级，但技术债务不还会拖慢后续迭代。

我的做法：每个迭代固定留 20% 时间给技术优化，避免债台高筑。同时把技术优化的收益翻译成业务语言——比如"启动速度优化 500ms 对留存的影响"，争取业务方理解。

### Q9: 怎么判断一个需求该不该做？

问三个问题：
1. 不做会怎样？——评估真实紧迫性
2. 做了能衡量效果吗？——避免"感觉有用"的需求
3. 有更简单的方案吗？——避免过度设计

如果三个问题都答不上来，这个需求就该砍。
```
