# 3D 球场（CocosTennis）— 面试叙述稿

## 第一段：背景与决策（30秒）

腾讯体育有大量非版权网球赛事，图文直播用户体验很差，但买版权成本太高。
我的方案是用 3D 动画直播平替图文直播。

关键技术决策：
- 引擎选 Cocos Creator 而非 Unity：轻量、支持 Web 导出、团队学习成本低
- 3D 资产用 AI 文生方案：降低美术资源生产成本
- 数据源用 Opta 实时数据：保证比赛数据准确性和时效性

## 第二段：关键架构设计（1分钟）

核心挑战是"数据驱动"——Opta 实时数据如何驱动 3D 球员行动和球路？

我设计的链路：
```
Opta 数据 → 数据解析层（标准化） → 动作映射层（数据→动画指令） → 3D 渲染层（Cocos Creator 播放）
```

关键设计点：
1. 动作映射层的抽象：比赛事件（发球/击球/得分）映射为动画状态机，支持不同球员的动作配置
2. 延迟控制：数据到达后必须在 500ms 内完成渲染，通过预加载动画资源和帧同步保证
3. 弹幕引擎定制：3D 场景内的弹幕需要和传统直播弹幕体验一致，我基于 Cocos 的渲染管线做了定制

备选方案及为什么不选：
- 用 Unity：太重，Web 导出体积大，不适合轻量场景
- 用视频流模拟 3D：交互性差，无法支持弹幕和视角切换

## 第三段：推动落地与结果（30秒）

这是我全链路 Owner 的项目：
- 协调方：服务端（数据接口）、数据组（Opta 接入）、客户端（Cocos 集成）
- 交付节奏：先做离线 Demo 验证数据驱动链路 → 接入服务端实时数据 → 客户端工程化优化
- 结果：交付了可用的非版权赛事 3D 直播方案，填补了"无版权赛事直播体验"的空白

## 追问预案

**Q: 最大的技术难点是什么？**
A: 数据驱动链路的实时性。Opta 数据不是为 3D 动画设计的，字段映射有大量 corner case。我的做法是先建立数据→动画的标准映射表，再通过离线回放大量历史数据做测试，逐步完善映射规则。

**Q: 如果重做，你会改什么？**
A: 3D 资产生产流程。AI 文生方案虽然降低成本，但质量不稳定。如果重来，我会建立一套标准化的资产生产管线，AI 辅助 + 人工审核，而不是纯 AI 生成。

**Q: 团队协作中遇到过什么冲突？**
A: 数据组提供的 Opta 接口字段频繁变更，导致映射层反复修改。我的解决方式是推动建立接口契约文档，约定字段稳定性承诺，同时在映射层加了一层适配器隔离变更影响。
