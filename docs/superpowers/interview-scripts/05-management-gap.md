# 管理短板回答稿 — "你没带过人，凭什么做负责人？"

## 核心回答（三段论，约2分钟）

### 第一段：诚实承认（20秒）

"我之前在腾讯体育更多是技术 Owner 角色，没有正式的管理职级和直接下属。但我认为小厂负责人的核心要求是'把一块业务撑起来'，而不是'管过人'。"

### 第二段：举证对等经验（60秒）

"虽然没有管理职级，但我做过这些负责人该做的事：

**技术选型与架构决策**：3D 球场的引擎选型（Cocos vs Unity）、PC 开播助手的技术栈决策（Electron + OBS Node）、AIRead SDK 的跨端方案选择（KMM vs Flutter/RN）。每个决策背后都有完整的技术调研和权衡过程。

**跨团队协调推动**：3D 球场涉及服务端、数据组、客户端三个团队的协调。我负责对齐各方的接口契约和交付节奏，保证全链路跑通。

**项目排期与风险管控**：PC 开播助手 2 个月从 0 到上线。中间遇到 OBS Node TRTC 改造的技术难点，我用最小可用版本策略——先验证核心链路，再逐步完善功能——保证即使延期也能交付核心价值。

**技术方案评审**：每个项目的关键架构设计都是我在主导——零拷贝渲染、状态机统一、协议改造。这些不只是'写代码'，而是'做正确的技术决策'。

**技术文档与知识沉淀**：我写了 17 篇 objc4 runtime 源码分析、16 章 AI Agent 算法拆解、团队 AI 工作流框架 OSC。这不是个人博客，而是把隐性知识显性化的能力——这也是负责人需要的能力。"

### 第三段：展示管理思路（40秒）

"如果让我带 5-10 人的客户端团队，我的思路很清晰：

1. **先理清业务目标和技术债务**，和上级对齐季度 OKR，把要做的事排优先级
2. **根据每个人的特长和成长意愿分配模块**，定期 1on1 对齐状态和困难
3. **建立代码评审和技术文档机制**，降低 bus factor——不能有人离职就带走关键知识
4. **我自己继续写核心代码**，以身作则——小厂负责人不能脱离代码，而且通过写代码能保持对技术细节的敏感度

管理技能可以在实践中学习，但技术判断力、项目推动力、系统性思维——这些我在过去几年的项目里已经证明了自己。"

## 追问预案

**Q: 如果团队有人不服你怎么办？**
A: 分两种情况。技术上的不服——用技术实力说话，我可以和他一起 review 代码、讨论方案，让他看到我的技术判断力。管理上的不服——我会主动了解他的诉求，是觉得我分配不合理、还是觉得我抢了他的机会。小团队的优势是可以直接沟通，不用绕弯子。关键是我要让团队看到：我是来帮他们成事的，不是来管他们的。

**Q: 你怎么做绩效评估？**
A: 三个维度——结果（交付了什么）、过程（怎么做的，有没有推动流程改进）、成长（个人能力有没有提升）。不和别人比，和上个季度的自己比。小团队不适合搞强制分布，核心是让每个人看到自己的成长路径。

**Q: 你打算怎么招人？**
A: 看三个东西——技术基本功（不是背八股，是解决问题的能力）、学习意愿（小厂技术栈变化快，不愿意学新东西的人很难跟上）、ownership（有没有"这是我的事"的意识）。面试题我会出实际业务场景的简化版，而不是纯算法题。招人的标准不是"比我强"，而是"能把分配给他的事做好且有成长空间"。
