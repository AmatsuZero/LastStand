# PC 开播助手（PCHelper）— 面试叙述稿

## 第一段：背景与决策（30秒）

业务痛点：
- 移动端开播屏幕小、拍摄角度受限，主播体验差
- 传统 PC 开播需要 OBS 运营人员配合，人力成本高

我的决策：
- 选 Electron + OBS Node 架构：Electron 保证跨平台和快速迭代（Vue.js 写 UI），OBS Node 复用 OBS 成熟的推流能力
- 关键创新：改造 OBS Node 支持 TRTC 推流——只修改协议类型（rtmp:// vs trtc://），最小化源码侵入，复用 OBS 原有 API

## 第二段：关键架构设计（1分钟）

最难的工程问题：跨进程视频渲染。

问题：Electron 的 Renderer 进程需要预览 OBS 画面，但 OBS 运行在 Main 进程。两个进程隔离，怎么把画面传过来？

我评估了三种方案：
1. **IPC 传帧**：把 Main 进程的 OBS 画面通过 IPC 逐帧传给 Renderer → 性能差，1080p 60fps 根本扛不住
2. **离屏渲染 + 共享纹理**：用 GPU 共享纹理跨进程 → 复杂度高，Electron 下兼容性差
3. **挖洞 + 同屏渲染**（我选的）：Renderer 进程在页面上预留一个透明区域（"挖洞"），Main 进程在这个区域上用原生窗口 + SDL 直接绘制 OBS 预览画面 → 零拷贝 GPU 渲染，性能最优

我的核心判断：宁可增加 C++/SDL 的复杂度，也不能牺牲视频预览的性能。这个方案需要处理 DPI 适配、渲染线程隔离、窗口生命周期管理等细节，但最终实现了流畅的 60fps 预览。

另一个关键设计：**SubRoom 多房间架构**
- 主播同时处于主直播间和虚拟陪看房间
- 通过 SEI（视频补充增强信息）将控制信息绑定到视频时间轴，做状态同步
- 保证了陪看房间的状态和主直播间严格同步

## 第三段：推动落地与结果（30秒）

从 0 到上线约 2 个月，涉及的技术栈：
- 前端：Electron + Vue.js
- 音视频：OBS Node C++ 改造 + TRTC 推流链路
- 原生渲染：SDL + 原生窗口管理

跨技术栈协调的挑战：前端、C++、音视频三个领域的人各自关注不同。我做了两件事：
1. 先画一张端到端架构图，让大家对"画面从哪来到哪去"有共识
2. 用最小可用版本（能推流能预览）快速跑通链路，再逐步完善功能

结果：项目上线后主播可以自主完成 PC 开播，不再依赖 OBS 运营人员。这是我最有成就感的项目，因为它从零开始，涉及多个技术领域的深度整合。

## 追问预案

**Q: 为什么不用 OBS 原生的虚拟摄像头方案？**
A: 虚拟摄像头方案无法做定制化——弹幕叠加、美颜滤镜、多场景切换这些需求都需要在 OBS 基础上做二次开发。OBS Node 给了我们完整的程序化控制能力。

**Q: Electron + 原生渲染的坑有哪些？**
A: 最大的坑是窗口层级管理。Electron 的透明窗口在 macOS 和 Windows 上行为不一致，macOS 上原生子窗口的 z-order 需要额外处理。另一个是 DPI 适配——高分屏下 SDL 渲染的坐标需要乘以 scale factor，但不同系统的 DPI 行为又不同。这些都是踩坑踩出来的。

**Q: 2 个月交付，中间有没有延期的风险？**
A: 有。OBS Node 的 TRTC 协议改造比预期复杂，因为 TRTC 的握手流程和 RTMP 差异大。我的做法是先做最小可用版本（能推流能预览），验证核心链路可行，再补周边功能。这样即使延期，核心价值也已经交付了。
