# 高频问题回答汇总

## 技术决策类

### Q1: 从零搭客户端团队，技术栈怎么选？

分三层看：
- **业务形态决定上层**：Native（高性能场景）、跨平台（多端一致）、小程序（流量入口）
- **团队能力决定中层**：KMM（有 Kotlin 基础）、Flutter（追求 UI 一致性）、RN（生态成熟）
- **基础设施决定底层**：CI/CD、监控体系、发布流程

我的经验矩阵：KMM 适合已有原生团队的跨端 SDK 场景，Electron 适合桌面端快速迭代，Cocos 适合 3D/游戏化场景。选型不是选"最好"的，是选"最匹配"的。

### Q2: 做过最难的架构决策？

PC 开播助手的跨进程视频渲染。Electron 下 Main 和 Renderer 进程隔离，OBS 画面在 Main 进程但需要在 Renderer 展示。

三个方案：
1. IPC 传帧：性能差，1080p 60fps 扛不住 → 不选
2. 离屏渲染 + 共享纹理：复杂度高，Electron 兼容性差 → 不选
3. 挖洞 + 同屏渲染：零拷贝 GPU 渲染，性能最优 → 选了

核心判断：宁可增加 C++/SDL 复杂度，不能牺牲视频预览性能。

### Q3: 跨平台方案怎么选？

三个维度：
- 性能要求：高性能 → Native / KMM；一般 → Flutter / RN
- 团队技能：Kotlin 基础 → KMM；Dart → Flutter；JS → RN
- SDK/库生态：特定平台能力（如 ARKit）→ Native

我的实践：AIRead SDK 选 KMM 因为体积敏感 + 团队有 Kotlin 基础。如果是业务 App 且追求 UI 一致性，Flutter 更好。没有银弹。

## 管理/推动类

### Q4: 项目延期怎么办？

三步走：
1. **判断根因**：需求变更 / 技术难题 / 人力不足 / 评估失误
2. **制定方案**：砍范围（降级非核心功能）/ 加人力 / 调排期
3. **透明沟通**：第一时间同步上级和相关方，不藏着掖着

我的实践：PC 开播助手 2 个月交付，OBS Node 改造比预期难。我用了最小可用版本策略——先验证推流+预览核心链路，再补周边功能。保证核心价值按时交付。

### Q5: 怎么处理技术分歧？

让数据和技术事实说话：
1. 列出方案 A 和方案 B 的优缺点、适用场景、风险
2. 如果有可量化的指标（性能、开发成本），用数据对比
3. 如果仍无法达成一致，我来做最终决策并对结果负责
4. 决策记录在文档里，后续可以复盘

我的博客习惯也体现了这一点——写下来，让事实可追溯。

### Q6: 你怎么做 1on1？

频率：两周一次，30 分钟。不谈进度（那是站会的事），谈三个问题：
1. 最近有什么让你开心/沮丧的事？
2. 遇到什么困难需要我帮忙？
3. 你想往什么方向成长？我能帮你做什么？

核心：1on1 是"他的时间"，不是"我的时间"。我听为主，说为辅。

## 业务 Sense 类

### Q7: 客户端团队怎么衡量价值？

三个维度：
- **业务指标**：留存、转化、用户时长——客户端直接影响用户体验
- **质量指标**：crash 率、ANR 率、启动时间、包体积——技术质量兜底
- **效率指标**：需求交付周期、发版频率——团队工程效能

我会定期向业务方同步这些数据，用数字说话，让技术价值可量化。

### Q8: 技术需求 vs 业务需求冲突？

先对齐优先级：业务需求通常更高优先级，但技术债务不还会拖慢后续迭代。

我的做法：每个迭代固定留 20% 时间给技术优化，避免债台高筑。同时把技术优化的收益翻译成业务语言——比如"启动速度优化 500ms 对留存的影响"，争取业务方理解。

### Q9: 怎么判断一个需求该不该做？

问三个问题：
1. 不做会怎样？——评估真实紧迫性
2. 做了能衡量效果吗？——避免"感觉有用"的需求
3. 有更简单的方案吗？——避免过度设计

如果三个问题都答不上来，这个需求就该砍。

## 反问环节

### 必问（展现 Owner 意识）

1. 客户端团队目前的规模和分工是什么样的？我进来后会负责哪块业务？
2. 团队目前最大的技术挑战或技术债务是什么？你对这个岗位的期望是什么？
3. 业务今年的核心目标是什么？客户端团队在其中的角色是什么？

### 选问（展现技术判断力）

4. 团队目前的技术栈是什么？有没有跨平台或技术升级的计划？
5. 团队的研发流程是怎样的？代码评审、测试、发布这些环节有没有痛点？
